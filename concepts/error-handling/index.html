Error Handling & Propagation

In traditional programming, handling errors is either polluted with implicit Exceptions (that could explode anywhere) or plagued by redundant if err != null checks (like in Go) or deeply nested .unwrap() calls.

Toka delivers a two-tier progressive approach to error logic, combined with the extremely concise ! propagation operator, bridging the gap between “static strict-type ironclad safety” and “smooth scripting-like coding experiences”.

Result and Option Containers

Section titled “Result and Option Containers”

Toka abolishes try...catch. Any output that can yield a failure or null must be explicitly housed inside of these wrappers:

  • Result<'T, 'E>: Contains the successful data 'T named Ok, or failing cause 'E named Err.
  • Option<'T>: Contains the expected data Some, or a legal nothingness None.

When you execute an IO operation that might fail, Toka’s compiler will strictly enforce that you must explicitly unwrap or destructure the container before you could touch the raw primitive. There is no silent NullPointerException:

! (Error Propagation) Operator

Section titled “! (Error Propagation) Operator”

When you have a long chain of operations with potential failure states, and your intention is to throw the error backwards to your caller immediately as it happens. Tearing it open brutally via .unwrap() is not only visually ugly, it also translates to fatal panic and abrupt application termination.

In Toka’s sugar-free paradigm, appending ! literally translates to an early & safe stack-return cutoff.

This is precisely why Toka shines brilliantly when handling Concurrencies and Networking architectures. Because it hinges on RAII semantic wrappers (like Vec buffers) tied to block execution scopes, when ! activates an early termination logic, any local resources allocated in the current scope will instantly undergo safe deallocation processes.

[!TIP] When to use !? When your application-level procedure defines a return type of Result<...>, go wild and spray ! across nested handlers!

If your function does not and cannot yield a Result constraint (for example main usually dictates i32, or if you never want system-level errors overflowing outwards), you should handle it conventionally by checking branches via res.is_err() and resolving them right there.
