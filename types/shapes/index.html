The Universal Shape

In the vast majority of programming languages, struct, tuple, union, and enum feature entirely different keywords and scoping rules. This introduces cognitive overhead and syntactic fragmentation.

Toka opts for the philosophy of Ultimate Simplicity: it abolishes all such disparate keywords, using the unified shape to define all structured data.

The compiler mathematically deduces exactly what kind of type you are building based on the format inside the parentheses, subsequently enforcing the appropriate memory layout and access safety constraints.

The Quadruple Automatic Deduction Table

Section titled “The Quadruple Automatic Deduction Table”
Syntax WrittenSystem Deduction KindMemory & Safety Constraints
shape Name(x: T, y: U)StructClassic product memory (contiguous). Direct access via .name is considered completely safe.
shape Name(T, U)TupleAnonymous product memory. Accessed via .0, .1 indices. Essentially a shorthand Struct.
shape Name(as T | as name: T)Union (Untagged)Types overlap in memory. Extremely dangerous; using . is strictly forbidden. Must use x as Type for memory-level reintepretation (Type Punning). Strictly prohibits holding pointers for safety.
shape Name(A | B(T) | C = N)Enum (Tagged)Equivalent to a Rust Enum. Automatically carries a discriminator tag. Mandates safe access and destructuring via match or if auto.

The @encap Trait (Privacy & Encapsulation Boundary)

Section titled “The @encap Trait (Privacy & Encapsulation Boundary)”

For a large complex shape, whether it be a Struct or Enum, it is strongly public to all files within the same package directory by default. But when facing cross-crate dependencies where you want to lock down core internal state from abuse, Toka’s sole and ultimate weapon is impl Name@encap.

Toka uses this elegant, tamper-proof encapsulation design to ensure that all unsafe access code must converge within the safe methods you expose (like m.lock()).
