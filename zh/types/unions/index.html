联合与枚举 (Unions & Enums)

在 Shape (形状) 章节中我们知道，当给 shape 增加竖线 (|) 分支时，它就脱离了普通积类型，转变成了联合类型（多个载荷共用内存）。

Toka 严厉区分这两者是带标签 (Tagged) 还是 无标签 (Untagged)，它们在访问安全控制上有着天壤之别。

1. 纯净枚举 Enum (带标签)

Section titled “1. 纯净枚举 Enum (带标签)”

最常见的场景。一旦任何一个分支仅仅是一个变体名，或者使用常规结构体格式 B(T) 定制负载，编译器会自动生成底层的标志位（Tag），将其提升为安全的 Enum。

必须使用 match 解构 (临时借用守卫)

Section titled “必须使用 match 解构 (临时借用守卫)”

Toka 出于内存安全，绝对禁止你使用 o.val 这样的点号访问 Enum 的成员！ 因为由于内存共享，如果你盲目访问时里面存的恰好是 None，就会立刻引发段错误崩溃。

2. 无标签重解释联合 Union (Untagged)

Section titled “2. 无标签重解释联合 Union (Untagged)”

当所有的分支都使用 as Type 或 as name: Type 这种重解释映射标记时，编译器将退化该结构为最危险但极其高效的底层 C-Union。此时系统不会分配额外的标志位。

这往往被广泛应用在与 C 侧网络开发交互、类型双关（Type Punning）的高端黑客魔法或者极致性能压缩场景。

极致危险：Union 的托管资源禁令 (E0801)

Section titled “极致危险：Union 的托管资源禁令 (E0801)”

出于防备灾难性的内存泄漏和悬垂解包，Toka 的语义分析器 (Sema) 严禁 在无标签的 Union 中存放任何带有生命周期管理的资源对象（例如 String、动态数组、独占指针 ^ 或带有自定义 @encap/drop 的结构体）。 因为在触发析构时，运行时无从知晓该 Union 当前激活的到底是哪一个变体，若放任编译器自动生成 drop 将导致致命段错误 (E0801)。如果您必须要在 Untagged Union 中存放托管资源，必须为其显式实现专属的 @encap 析构函数来接管生命周期。否则，请老老实实使用带标签且能通过 match 被安全释放的正规 Enum。
