Toka 为现代多核时代的并发编程提供了极其严谨的编译期内存屏障。通过结合指针形态学的所有权机制与标准库的 `@encap` 数据结构封装，Toka 使得在没有冗长繁复生命周期标注的情况下，依然能够写出极具安全性的并发代码。

## 线程分发机制 (Thread Spawning)

Toka 利用操作系统级别的本地线程 (Native Threads) 执行物理级并行处理。开启新线程只需调用 `thread_spawn`，并传递一个带有 `cede` 转移所有权的闭包任务：

```toka
import std/thread::{thread_spawn, sleep_ms}

pub fn main() -> Result<i32, String> {
    // 启动线程并期待其返回一个 i32 类型结果
    auto handle# = thread_spawn<i32>({ =>
        println("来自分离线程的问候！")
        return cede 0
    })
    
    // 主线程等待它执行完毕。如果线程崩溃抛出错误，! 将往上传播
    handle#.join()!
    return Result<i32, String>::Ok(0)
}
```

## 互斥锁与 RAII 安全 (Mutex & Guard)

Toka 由于废弃了传统指针而使用安全封装体系，它的互斥锁 `Mutex<T>` 是将**数据和锁紧紧绑定在一起的**（基于数据封装而不是锁代码段）。你不可能在未解锁的情况下接触到被锁的数据。

这一切通过基于 RAII（资源获取即初始化）和 作用域隐式析构 来完成：

```toka
import std/sync::{Mutex}
import std/thread::{thread_spawn}
import std/io::{println}

pub fn main() -> Result<i32, String> {
    // 将变量 0 放入一个包裹着 i32 的互斥锁
    auto m# = Mutex<i32>::new(0)
    
    // 大括号构造明确的作用域生命周期
    {
        // 1. 尝试获取锁，利用 ! 直接将任何上锁期间可能存在的毒化异常向外抛射
        auto mut_guard = m#.lock()!
        
        // 2. 此时，你的手中的 mut_guard 是数据的唯一安全守卫。
        // 使用它提供的方法，获取内部数据实体本身的指针控制器！加上 # 获得读写权。
        auto *mut_val# = mut_guard.get_ref()
        
        // 3. 剥去指针帽子，即可写入数据本体！
        mut_val = mut_val + 10
        println("我把值修改成了: {}", mut_val)
        
    } // <-- 大括号结束！隐式的 drop 会析构 mut_guard 这个守卫，进而自动释放锁
    
    return Result<i32, String>::Ok(0)
}
```
**安全防御核心**：在 Toka 标准库设计中，由于实施了 `@encap` 特征隐蔽，`m.data` 对于外部代码绝对不可见！开发者被**强制约束**只能通过 `m#.lock()` 方法拿到生命周期守卫，彻底告别了“忘了加锁却读写了数据”的危险。

## 并发类型标记

与 Rust 一脉相承但更为简洁，Toka 用特征标记防御非法类型跨越线程的深渊：
- **`@Send`**: 暗示某数据可以直接穿越虫洞交给另一个线程。（例如所有不带指针的值类型均自动是 Send）
- **`@Sync`**: 暗示某数据的引用如果在多个线程间同时被访问也是安全的。（例如加了壳的 `Mutex` 类型会自动获得该加持）。
