---
title: 接口与特征 (Traits)
description: Toka 中的鸭子类型与系统级的硬性封装屏障。
---

Toka 提供了一套极为轻量化但却掌控着系统最底层内存安防的体系：**Trait**。它不需要繁杂的申明，往往只有在调用或是要求极高约束时才会显式登场。

## 隐式鸭子类型 (Implicit Duck Typing)

Toka 目前（截止 1.3）并未采用非常严格的类似于 Rust 面向所有对象的 `trait XXX { fn yyy(); }` 声明格式。只要你在代码里调用了一个类型的方法，比如 `p.draw()`，Sema 会直接探测 `p` 背后是否有通过 `impl P { pub fn draw() }` 绑定的同名固有实现。如果有，即通过编译；如果没有则报错。

这意味着绝大多数情况下，你不需要提前写一个冗长的 `Drawable` 借口再去一一映射。

## 内置约束 Trait (Bound Traits)

为了让泛型系统的推导更为安全，或者在系统级别对接运行时，Toka 内含了一批具有特定语义的编译器级别的特殊 Trait。在使用泛型 `T` 时可以进行**显式约束**：

```toka
// 约束 'K 必须实现 Hash 特征（即其本质是可以被哈希的类型）
shape HashMap<'K: @Hash, 'V> {
    // ...
}
```

常见的内置 Trait 包括但不限于：
- `@Hash`: 类型必须存在固有的 `.hash()` 方法或者系统已为其内置。
- `@PartialEq`: 类型必须支持 `==` 运算符。
- `@Sync` / `@Send`: 类型具有线程并行传递或数据读取的安全性证明。
- `@ToString`: 基础字符串转换契约，要求类型必须实现 `to_string(self) -> String`，以用于普通 `{}` 打印。
- `@ToFormat` (新特性): 高级格式化契约，要求类型必须实现 `to_string_fmt(self, fmt: view_str) -> String`，以配合 `println` 智能格式化语法（如 `{:04}`、`{:.2}` 等形式）进行精细格式控制。

## 字符串转换与格式化特征 (@ToString 与 @ToFormat)

Toka 拥有一套极其现代且安全的 `{}` 占位符字符串格式化与打印体系，彻底拒绝了传统的 C 风格 `%` 格式化符号。该体系依托以下两个标准库的内置契约运作：

### 1. `@ToString` 协议
用于普通的 `{}` 无参数占位符插值。
- **要求**：类型固有实现 `to_string(self) -> String`。
- **原理**：当你在 `println("val: {}", val)` 中打印非字符串数据时，语义分析器 (Sema) 与 CodeGen 会自动搜寻该类型的 `@ToString` 固有特征。
- **零拷贝特化**：若插值目标本身为 `String` 或 `view_str`，Toka 将会强力降级为高效的**零拷贝（Zero-Copy）**投影写入，彻底避免无意义的二次克隆与堆分配。

### 2. `@ToFormat` 协议
用于带有冒号格式控制符的 `{:指令}` 占位符打印。
- **要求**：类型固有实现 `to_string_fmt(self, fmt: view_str) -> String`。
- **原理**：当你使用诸如 `{:04}`（四位补零）、`{:x}`（十六进制）或 `{:.2}`（浮点数两位精度）等高级指令时，编译器会强行验证目标类型是否实现 `@ToFormat`。若验证通过，CodeGen 会在编译期将 `:` 后的指令自动切片打包为临时 `view_str`，零成本地注入作为 `to_string_fmt` 的第二个参数，并依靠底层 C 桥接层与内存缓冲区完成最终的高性能格式渲染。
- **自定义 Shape 格式化**：您可以为任意自定义的 Shape 实现 `@ToFormat` 特征，从而自主解析复杂的指令（例如让您的数据结构响应 `{:json}` 指令输出 JSON 串）。

## @encap：防篡改的心脏

在绝大多数面向对象语言中，你可以通过 `private` 和 `public` 肆意修饰各个字段来管理可见性。但在 Toka 中，一切字段如果没有外部标记本身就是私有的，我们通过一个**终极武器 `@encap` Trait 来声明模块级的铁幕**。

当你给自己的复杂系统编写了一个核心的抽象类型（例如 `Mutex`），你绝对不希望使用者绕过死锁检测，直接通过 `mutex.data` 去读取裸数据。

**使用 `@encap` 实现强力锁死：**

```toka
shape Mutex<T> (
    data: T,
    handle: u64
)

impl<T> Mutex<T>@encap {
    // 此处留空即代表最高级保密！任何人均不能通过 m.data 访问 data
}

impl<T> Mutex<T> {
    pub fn lock(self#) {
        // 我只把锁公开！
    }
}
```

如果你在标准库编写跨多个内部库的基础组件，你可以**打开白名单漏洞**：

```toka
impl<T> TaskQueue<T>@encap {
    pub(crate) *   // 当前项目空间下所有文件畅通访问全部成员！
    pub(std) *     // 对标准库 `std` 层级大开绿灯！
}
```

Toka 在最新的语义构建中，连同泛型子实例化对象的特征一同牢牢受限于 `@encap`，任何跨越授权路径层级的非法字段调用都会立即报出 `Member xxx of shape yyy is private` 的红牌拦截。
