随着你的 Toka 应用或第三方库不断扩大，保持一个清晰、直观的目录结构变得至关重要。虽然 Toka 编译器本身不会严格限制你的文件夹如何布局，但遵循社区推崇的**最佳实践约定**，能确保你的代码具备高可维护性，并让其他开发者能够丝滑地接入你的代码。

## 1. 可执行项目 (应用/工具)

如果你正在编写一个最终会被编译为二进制可执行文件的应用程序，你的入口点永远应该是 `src/main.tk`。

### 标准项目布局

```text
my-app/
├── Project.tk        # 构建系统配置与依赖声明
├── src/
│   ├── main.tk       # 绝对入口，必须包含 pub fn main() -> i32
│   ├── utils.tk      # 应用特定的辅助逻辑
│   └── models/
│       ├── user.tk
│       └── session.tk
└── tests/            # 集成与单元测试
    └── pass/
```

### 最佳实践建议
- **专属入口**：始终将你的 `pub fn main()` 放在 `src/main.tk` 中。
- **模块细分**：随着应用变大，应当在 `src/` 内部建立合理的子文件夹（如 `src/models/`，`src/services/`）来进行解耦。

## 2. 第三方库项目 (Library)

如果你正在开发一个供他人导入使用的第三方库，代码组织的**第一核心目标是：提供极其清晰的 Import 体验**。

> [!WARNING]
> **反模式 (Anti-Pattern)：过度碎片的假微服务布局**
> 绝对不要为库中的每一个组件都建立一个极深的文件夹，并且在里面塞满 `main.tk`。
> 
> **错误示范：**
> ```text
> lib/
> ├── mylib_button/
> │   └── main.tk    ❌ 库文件绝对不要命名为 main.tk！
> ├── mylib_chart/
> │   └── main.tk    ❌ 导致外部导入极度恶心：import lib/mylib_chart/main
> ```
> 这种“伪微服务式”的布局不仅让目录树极度碎片化，还会让使用者的 `import` 语句变得冗长且毫无逻辑。

### 标准库布局

正确的做法是：**扁平化你的组件，并将它们统一收纳在一个专属于你库名的命名空间（文件夹）下。**

```text
my-lib/
├── Project.tk
├── lib/
│   └── mylib/           # 你的专属命名空间文件夹
│       ├── button.tk    # 清爽、直白的组件文件名
│       ├── chart.tk
│       └── colors.tk
└── tests/
```

### 最佳实践建议
- **严禁库代码使用 `main.tk`**：在 Toka 与大多数现代语言中，`main` 这个名字暗示了它是可执行程序的入口点（Entry Point）。库内部的模块必须直接命名为其代表的含义（如 `button.tk`）。
- **统一的命名空间**：永远把你的代码放在 `lib/` 下的一个专属子目录里（如 `lib/mylib/`）。**绝对不要**把零散的 `.tk` 文件直接暴露在根 `lib/` 目录下，这会严重污染使用者的包管理空间。
- **优雅的导入体验**：遵循上述标准布局后，你的使用者就能写出非常优雅、直觉的导入语句：
  ```toka
  import lib/mylib/button::{render_button}
  import lib/mylib/chart::{draw_chart}
  ```

## 3. 通用约定

- **文件后缀**：所有的 Toka 源代码文件都必须使用 `.tk` 后缀。
- **蛇形命名法 (Snake Case)**：无论是文件夹名还是文件名，都应当使用 `snake_case`（例如 `file_reader.tk`, `network_client/`）。
- **常量剥离**：如果你的库定义了大量的硬编码常量（例如终端的 ANSI 颜色控制码、大量数学常量），请务必将它们剥离到一个独立的文件中（如 `colors.tk` 或 `constants.tk`），不要将它们与核心业务逻辑混写在一起。
