Toka 结合了 **所有权 (Ownership)** 和 **RAII (资源获取即初始化)** 机制，在确保内存安全的同时实现零开箱开销。

## 单一所有者原则

对于任何堆分配的资源（通过 `new` 创建），如果使用了 `^` (唯一所有权) 指针，Toka 将强制执行严格的单一所有者规则：

1. **谁持有谁负责**：持有 `^T` 唯一指针的作用域就是该资源的“所有者”。
2. **确定性析构**：当该作用域结束时，内存会立即且确定性地被释放。
3. **默认移动 (Move)**：将 `^T` 传递给函数或赋值给其他变量会转移所有权。原始变量随后将变为无效状态。

```toka
fn main() {
    auto ^p = new i32(42) // p 是所有者
    {
        auto ^q = p       // 所有权转移到了 q。p 现在是无效的！
        println("值: {}", *q)
    } // q 离开作用域 -> 内存在此处被释放！
    
    // println("值: {}", *p) // 编译错误：p 的值已被移动
}
```

## 自动合成 Drop 函数

Toka 编译器会自动为所有拥有的类型合成 `drop` 调用：
- 对于 **Shapes**：它会递归地调用所有字段的 `drop`。
- 对于 **集合类型** (如 `Vec<T>`)：它会在释放自身缓冲区前先销毁所有元素。

### `@encap` Trait

对于需要自定义清理逻辑的资源（如文件句柄、套接字或互斥锁），可以通过实现 `@encap` Trait 来定义。这告诉编译器在资源不再需要时应如何正确关闭或清理它。

```toka
shape File (
    #handle: i32
)

impl @encap for File {
    fn drop(self) {
        // 自定义关闭 OS 句柄的逻辑
        libc::close(self.handle)
    }
}
```

## RAII 的实战价值

得益于 RAII 与 **指针构词学** 的深度整合，你可以安全地混合使用多种内存管理模式：

- **局部生命周期**：使用 `^T` 对堆内存进行类栈式的自动管理。
- **共享数据**：使用 `~T` 进行引用计数共享（适用于图状结构）。
- **临时访问**：使用 `&T` 将数据传递给函数，无需承担所有权转移或引用计数的开销。

## 内存安全保证

Toka 的语义分析器 (Sema) 在编译期拦截常见的内存错误：
- **移动后使用 (Use-After-Move)**：在编译阶段全程跟踪所有权。
- **悬空引用 (Dangling References)**：借用检查器确保引用不会比它的所有者活得更久。
- **重复释放 (Double-Free)**：单一所有者机制使得这类错误在逻辑上不可能发生。
