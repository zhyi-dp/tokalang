在 Toka 中，每个建立堆空间的复合对象都由两个截然不同的层面构成：它的 **Soul (灵魂/数据本体)** 和它的 **Handle (壳/指针/操纵杆)**。掌握如何准确命中这两个层面，是你学习 Toka 时最重要的一课。

## Soul (灵魂 / 值本身)
“Soul” 代表对象的实际数据块、结构体成员或内存状态。当你对 soul 执行操作时，你在直接修改这块内存里的字节。

- **感知方式**: 任何不带指针符号出场的变量名，比如 `p`。
- **可变性标识**: 在变量名后追加 `#` 即表示 Soul 是可变的（允许被赋值写入），如 `auto x# = 10`。
- **操作影响**: `p = q` 或者 `p.name = "Toka"` 会直接变更内部存储的数据。由于数据往往是庞大的，默认的赋值（当并非简单值类型时）会隐式产生深层次拷贝或触发 `cede` 的移动所有权检测。

## Handle (壳 / 操纵杆 / 指针实体)
“Handle” 是你在栈上握持的那个操纵杆，它记录着对象在堆内存中的确切坐标以及管理权限（所有权类型）。当你操作 Handle，你是在“拔掉插头重新插在别处”。

- **感知方式**: 在变量名前追加形态符号（如独占的 `^p`，共享的 `~p`，或是原始指针 `*p`，借用的 `&p`）。
- **可重绑定性标识**: Handle 自身是否可以更换指向？这由 Handle 定义时的 `#` 决定。比如：`auto ^p# = new Node()` 代表连这个指针本身也是可变的，它可以被重写指向另一个地址。
- **操作影响**: `^p = ^q` （假设是指针重绑定）不会对 `q` 或 `p` 最初指向的实际结构体数据做哪怕一个字节的修改。它只做一件事：抛弃 `p` 也就是原本的控制权（触发释放），然后把 `q` 代表的控制权接驳上来。对于独占指针 `^`，伴随重绑定的通常是隐式 `move` 语义（使用 `cede` 显式交接）。

## “戴帽”法则与深浅赋值的终结

许多现代编程语言由于采取了隐式的引用机制（例如 Python, Java, JS 等），模糊了“我要交换这两个引用的指向”还是“我要把那份数据克隆过来”的操作。

在 Toka 中，这种二义性被彻底消除：

1. **`p = q`** 
   - **灵魂操作 (Soul Operation)** 
   - 目标：把 `q` 的灵魂赋值并写入 `p`。

2. **`^p = cede ^q`**
   - **载体操作 (Handle Transference)**
   - 目标：将 `q` 指针所拥有的一切权力和目标地址都转移给 `p`，这涉及原 `p` 目标的销毁。

| 概念 | 访问途径 | 实际代表什么 | 常见操作 | 可变性控制 |
| :--- | :--- | :--- | :--- | :--- |
| **Soul (灵魂)** | 裸写 `p` | 真实堆/栈内存数据 | `p.val = 1`, `print(p)` | 用 `auto p#` (或 `*p#` 若其是个指针) 开启数据体层面的 `#` 可写权限。 |
