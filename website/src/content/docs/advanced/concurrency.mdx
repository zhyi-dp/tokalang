Toka provides an extremely strict compile-time memory barrier for concurrent programming in the multi-core era. By marrying the Ownership mechanics of Pointer Morphology with the Standard Library's `@encap` data structure encapsulation, Toka allows you to write highly secure concurrent code without any verbose lifetime annotations.

## Thread Spawning

Toka leverages OS-level Native Threads for physical parallel processing. To spawn a new thread, simply call `thread_spawn` and pass a closure task that transfers ownership via `cede`:

```toka
import std/thread::{thread_spawn, sleep_ms}

pub fn main() -> Result<i32, String> {
    // Spawn a thread expecting it to return an i32 result
    auto handle# = thread_spawn<i32>({ =>
        println("Hello from a detached thread!")
        return cede 0
    })
    
    // The main thread waits for it. If it panicked or crashed, ! throws the error outwardly.
    handle#.join()!
    return Result<i32, String>::Ok(0)
}
```

## Mutex & RAII Safety (Guard)

Because Toka abolishes traditional pointers in favor of a safe encapsulation hierarchy, its `Mutex<T>` **tightly binds the data and the lock together** (based on data encapsulation rather than code-block locking). It is impossible for you to touch the secured data without explicitly unlocking it.

This is achieved via RAII (Resource Acquisition Is Initialization) and implicit scope destruction:

```toka
import std/sync::{Mutex}
import std/thread::{thread_spawn}
import std/io::{println}

pub fn main() -> Result<i32, String> {
    // Pack the variable 0 into a Mutex enveloping an i32
    auto m# = Mutex<i32>::new(0)
    
    // Curly braces establish an explicit scope lifetime
    {
        // 1. Attempt to acquire the lock. Leveraging `!` throws any poisoning exceptions instantly outwardly!
        auto mut_guard = m#.lock()!
        
        // 2. Now, the `mut_guard` in your hands is the sole safe defender of the data.
        // Use the methods it provides to get the pointer controller of the internal data entity itself! 
        // Add # to gain read/write permissions.
        auto *mut_val# = mut_guard.get_ref()
        
        // 3. Take off the pointer hat to mutate the data directly!
        mut_val = mut_val + 10
        println("I modified the value to: {}", mut_val)
        
    } // <-- Ends curly brace! The implicit drop destructs `mut_guard`, which in turn automatically releases the lock.
    
    return Result<i32, String>::Ok(0)
}
```
**The Core of Safety Defense**: Within the Toka Standard Library design, because `@encap` trait concealment is enforced, `m.data` remains absolutely invisible to external code! Developers are **compulsorily bound** to go through the `m#.lock()` method to retrieve the lifetime guard, bidding a final farewell to the dangerous "forgot to lock but read/wrote data anyway" scenario.

## Concurrent Type Markers

Inheriting the spirit of Rust but more concise, Toka uses trait markers to guard against illegal type straddling across the threading abyss:
- **`@Send`**: Implies a piece of data can safely traverse the wormhole and be handed to another thread. (For instance, all pure value types without pointers are automatically Send).
- **`@Sync`**: Implies that references to a piece of data are safe to be accessed simultaneously by multiple threads. (For instance, wrapping a type in a `Mutex` shell automatically grants this blessing).
