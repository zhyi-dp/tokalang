title: Pointer Morphology
description: Unique, shared, and borrowed pointers in Toka.
---

Toka's **Pointer Morphology** defines the nature of an object's identity and ownership through concise sigils attached to the variable or field name.

## Pointer Types (Sigils)

Toka uses three primary sigils:

### 1. Unique Pointer (`^`)
Represents exclusive ownership. When a `^T` goes out of scope, the memory is automatically freed (RAII).

```toka
auto ^p = new i32(10)
```

### 2. Shared Pointer (`~`)
Represents shared ownership using thread-safe reference counting.

```toka
auto ~p = new i32(20)
auto ~q = ~p // Reference count increases
```

### 3. Borrowed Pointer (`&`)
Represents non-owning access to data owned by someone else.

```toka
auto x = 10
auto &p = &x
```

## Handle Attributes

Pointers themselves (the Handles) can yield additional control through prefixes.

### Pointer Nullability (`nul`)
The `nul` prefix allows a pointer to be `null`.

```toka
auto nul ^p: User = null
```

### Handle Rebindability (`#`)
Adding `#` after the sigil allows the pointer variable to be reassigned.

```toka
auto ^#p = new i32(1)
^p = ^q // OK: Handle is rebindable
```

## The "Chemical Formula" Stacking

You can stack these dimensions on any pointer.

```toka
auto nul ^#node#: Node? = null
```

- `nul`: Pointer is nullable.
- `^#`: Pointer is unique and the handle is rebindable.
- `node#`: The pointed-to data (Soul) is mutable.
- `Node?`: The entity itself can be `none`.

## Automatic Dereferencing

Toka follows the **Hat Principle**: if you access the variable name `p` directly, you are accessing the **Soul** (the data). The compiler automatically dereferences the pointer for you.

```toka
auto ^p = new User(name = "Alice")
println(p.name) // Automatically dereferences p to access name
p.name = "Bob" // Automatically dereferences p to mutate name (if User is mutable)
```

## Lifetime Effect Routing (`effects:`)

Instead of relying on sprawling generic lifetime parameters (`<'a>`), Toka utilizes the `effects:` block alongside **named return values** to establish explicit pointer borrowing and ownership maps (directed edges).

```toka
fn compose(a: i32, b: i32) -> &res: Pair
effects:
    &res <- a | b
{
    // The lifetime of res is automatically constrained to the intersection of a and b
}
```
Through the `<-` syntax sigil, the compiler can precisely trace the origin and lifetime constraints of borrowed pointers across complex boundary conditions.
