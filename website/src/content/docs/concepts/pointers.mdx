title: Pointers & Hat Principle
description: Toka's four core pointer morphologies and the profound philosophy of Soul vs Handle separation.

Instead of adopting the traditional asterisk (`*`) and ampersand (`&`) combinations seen in C/C++ or Rust, Toka completely reimagines memory identity semantics by introducing the **Hat Principle**.

At the core of Toka's design philosophy lies the maxim: **"Morphology is Identity, Unhatting reveals the Soul".**

## I. The Four Hats

Toka strictly prohibits implicit pointer operations or hidden references. Whenever a pointer variable appears in your code, it MUST wear its morphological sigil (its "Hat"), otherwise it will trigger an automatic dereference (unhatting).

| Sigil | Morphology | Semantics & Use Cases |
| --- | --- | --- |
| **`^`** | **Unique Pointer** | Holds absolute ownership. Automatically destructed (RAII). Assignment implicitly triggers a `move`, stealing ownership. Ideal for root objects in managed memory. |
| **`~`** | **Shared Pointer** | Shared ownership via built-in reference counting. Increments on assignment. Common for cross-thread data or graph node management. |
| **`*`** | **Raw Pointer**    | Manually handles memory addresses. Carries no automatic management or ownership. Must be operated within an `unsafe` context. |
| **`&`** | **Reference**      | A borrowed observer with a strict temporal scope. **It does NOT mean "address-of"**. It safely freezes the target owner's mutate permissions until the observation period expires. |

> [!WARNING]
> Unlike the C family, Toka **strictly forbids** stacking hats in front of variable names. Syntax like `**p` or `^&p` is completely illegal.

### Morphology Belongs to the Variable, Not the Type

A defining and powerful characteristic of Toka: **Pointer sigils are always worn by the variable name, never by the type name.** (This is why they are called "Hats").

```toka
// Correct (Toka Style)
auto ^p: Node = new Node() 

// Incorrect (Traditional C/Rust mindset; the compiler will reject this)
auto p: ^Node = new Node()
```

## II. Separation of Soul and Handle

Why must the pointer sigil precede the variable? It stems from Toka's cardinal rule:

- **Wearing the Hat (`*p`, `^p`, `~p`, `&p`)**: You are touching the **Handle** itself. Altering it means **rebinding the pointer's destination**.
- **Taking off the Hat (`p`)**: You are touching the **Soul** (the actual data). Operating on it means **directly reading/writing the target memory**.

### A Subversive Reverse Mindset

Developers coming from C/C++ should note: Toka's dereferencing logic is the exact mirror image of C:

**In C++:**
- `p` is the pointer Handle.
- `*p` is the dereferenced data.

**In Toka:**
- `*p` (or `^p` / `~p`) is the pointer Handle.
- `p` is the dereferenced data (Soul).

### Code Example: Identity-Conserving Assignment

Because morphology represents identity, the Hat status on both sides of the assignment operator must be visually symmetrical:

```toka
auto ^p = new i32(42)

// 1. Handle Operation (Rebind pointer destination)
auto ^q = new i32(100)
^p = cede ^q     // Swap control while wearing hats! ^p gains ownership of ^q, original ^p data is dropped.

// 2. Soul Operation (Read/write final data)
p = 200          // Take off the hat, write directly into p's memory (auto-dereference).
```

> [!NOTE]
> This means whenever you see an unhatted bare name `p` in the code, you can confidently operate on it and access its fields (e.g., `p.x`) just like a normal value type. The compiler has already safely dereferenced and traced the pointer for you behind the scenes.

## III. Dimensional Reduction & Borrowing

### 1. Safe Borrow Projection (`as`)
To safely observe data owned by a `^` or `~`, use a safe borrow projection. 
Assuming `^q` exists:

```toka
// 'as' acts as a safe projection operator
auto &p = ^q as &Node
```
In this instant, `^q` enters a FROZEN state. Until the observer `&p`'s lifetime concludes, no one can modify the original struct.

### 2. OAddr: Universal Observation Lens
If you simply need a numeric memory address for logging or C-FFI interfacing, without involving any lifetime semantics:

```toka
// Obtain the raw pointer's numeric address as usize for printing
println("Pointer address: 0x{:X}", *raw as OAddr) 
```
