---
title: Soul vs Handle
description: Uncover the profound storage philosophy inside Toka structures, understanding the difference between data mutation and lifecycle resetting.
---

In Toka, every complex object allocated on the heap is composed of two entirely distinct layers: its **Soul (the data entity)** and its **Handle (the pointer/joystick)**. Accurately targeting these two layers is the most crucial lesson in learning Toka.

## The Soul (The Data Itself)
The "Soul" represents the actual block of data, struct members, or memory state of an object. When you operate on the soul, you are directly modifying the bytes in that memory block.

- **Perception**: Any variable name appearing without a pointer sigil, such as `p`.
- **Mutability Tag**: Appending `#` after the variable name signifies the Soul is mutable (open for assignment), e.g., `auto x# = 10`.
- **Operation Impact**: `p = q` or `p.name = "Toka"` directly mutates the internally stored data. Because data is often bulky, default assignments (when not simple primitives) will implicitly trigger deep copies or invoke `cede` move semantics.

## The Handle (The Joystick / Pointer Entity)
The "Handle" is the joystick you hold on the stack. It records the exact coordinates of the object in heap memory alongside its access privileges (ownership type). When you operate on the Handle, you are "unplugging the cable and plugging it elsewhere".

- **Perception**: A morphology sigil prepended to the variable name (such as the exclusive `^p`, shared `~p`, raw pointer `*p`, or borrowed `&p`).
- **Rebindability Tag**: Can the Handle itself change where it points? This is determined by the `#` during Handle declaration. For example: `auto ^p# = new Node()` means the pointer *itself* is mutable and can be rebound to a different address.
- **Operation Impact**: `^p = ^q` (assuming a pointer rebind) will not alter a single byte of the actual struct data originally pointed to by `q` or `p`. It does only one thing: it discards `p`'s original ownership (triggering deallocation), and patches the control from `q` into `p`. For an exclusive pointer `^`, rebinding is usually accompanied by implicit `move` semantics (explicitly handed over via `cede`).

## The "Hat" Principle and the End of Deep/Shallow Ambiguity

Many modern programming languages obscure whether an operation means "I want to swap where these references point" or "I want to clone the data over" due to implicit reference mechanisms (like in Python, Java, or JS).

In Toka, this ambiguity is completely eradicated:

1. **`p = q`** 
   - **Soul Operation** 
   - Goal: Copy and write the soul of `q` into `p`.

2. **`^p = cede ^q`**
   - **Handle Transference**
   - Goal: Transfer all powers and addresses owned by `q` to `p`. This involves destroying whatever was originally at `p`.

## Quick Reference Cheat Sheet

| Concept | Access Method | What it Represents | Common Operations | Mutability Control |
| :--- | :--- | :--- | :--- | :--- |
| **Soul** | Bare `p` | Actual heap/stack memory data | `p.val = 1`, `print(p)` | Use `auto p#` (or `*p#` if a pointer) to unlock `#` write-access to the data body. |
| **Handle** | Hatted `^p`, `*p` | Stack-based address/ownership controller | `^p = ^q`, `as &p` for observation | Use `auto ^p#` (to control whether the pointer itself can be rebound). |
