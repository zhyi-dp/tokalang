title: Ownership & RAII
description: Resource management and memory safety in Toka.
---

Toka's memory safety is built on a clear **Ownership** model, enforced by the compiler through Pointer Morphology and the innovative **PAL (Path-Anchored Ledger)** borrow checker.

## Unique Ownership (`^`)

A Unique Pointer (`^T`) is the sole owner of a resource. When it goes out of scope, the resource is automatically cleaned up (RAII).

```toka
fn take_ownership(cede ^u: User) {
    println("Owning user: {}", u.name)
} // u is dropped here

pub fn main() -> i32 {
    auto ^user = new User(name = "Alice")
    take_ownership(cede ^user) // ownership transferred with cede
    // ^user is no longer accessible here
    return 0
}
```

## Shared Ownership (`~`)

Shared Pointers (`~p`) use reference counting. The resource is freed only after the last reference is dropped.

```toka
shape Node (
    value: i32,
    ~next: Node?
)
```

## Borrowing (`&`)

Borrowing (`&p`) allows you to access data without taking ownership. Borrows can be immutable (`&p`) or mutable (`&p#`).

```toka
fn print_user(&u: User) {
    println(u.name)
}

pub fn main() -> i32 {
    auto ^user = new User(name = "Bob")
    print_user(&user) // Immutable borrow (implicit capture)
    return 0
}
```

## Explicit Transfer (`cede`)

You can explicitly transfer ownership of a Soul using the `cede` keyword.

```toka
fn process_data(cede ^v: [i32]) { ... }

pub fn main() -> i32 {
    auto ^v = [1, 2, 3]
    process_data(cede ^v) // ^v is ceded
    return 0
}
```

## The @encap Trait

For types that manage external resources (like file handles or sockets), implementing the `@encap` trait allows for custom cleanup logic.

```toka
shape File (
    handle: i32
)

impl File@encap {
    pub fn drop(self) {
        // Custom close logic
    }
}
```
