---
title: Ownership & RAII
description: Resource management, lifetimes, and the `cede` transference mechanism in Toka.
---

Toka's memory safety is built upon an extremely rigorous yet developer-friendly **Ownership** model. Unlike Rust's verbose lifetime annotations, Toka frees you from the cognitive burden by relying on the "Hat Principle" and a stealthy background tracker to manage the life and death of all your resources.

## Resource Binding and Deallocation (RAII)

In Toka, when you allocate managed heap memory via `new`, or declare a struct that incorporates allocations, you become the owner of that resource. When the owner (typically an exclusive pointer `^` or a stack variable) leaves its scope, the compiler automatically invokes its `drop` destructor function.

```toka
fn process() {
    auto ^user = new User(name = "Alice")
    // ... joyful operations
} // <-- Leaves scope! The heap memory and object managed by ^user are safely and automatically freed.
```

## Function Boundaries: The Default Capture Rule

Unlike other systems languages, **function parameters in Toka are implicitly "captured" by default (observing the source without borrow annotations)**.
Passing parameters does NOT spontaneously trigger any Copy or Move operations. This means the arguments you pass to other functions will still be alive and well after the function call concludes.

```toka
fn print_user(u: User) {
    println(u.name)
}

pub fn main() -> i32 {
    auto ^user = new User(name = "Bob")
    print_user(user) // The parameter 'u' only briefly observes 'user'.
    
    // ^user is still usable!
    println("User is still alive: {}", user.name)
    return 0
}
```

## Explicit Transference (`cede`)

If you truly need to hand over ownership to someone else (e.g., storing it into an array, or sending it to a separate thread), because the default is safe observation, you must employ the `cede` keyword for **explicit transference**. Once ceded, the original variable is "drained"; subsequent attempts to access it will instantly trigger a compilation error.

### 1. Transference via Assignment
```toka
auto ^a = new i32(42)
auto ^b = cede ^a  // ^a surrenders its resources to ^b. Afterwards, ^a becomes invalid.
```

### 2. Partial Move Restrictions (The Breakdown Ban)
**You cannot forcefully strip resources out of a managed struct.** If a struct encapsulates its own lifecycle (has an `@encap` or manages memory), you are strictly forbidden from using `cede` on its internal fields (e.g., `auto x = cede my_map.inner_tree`). Doing so would shatter the struct's safe destructor chain, leaving the host object in a catastrophic, unrecoverable "partially moved" state. You must `cede` the entire host object, or borrow its contents.

### 3. Transference Across Function Boundaries
If a function signature specifically requires taking over a resource, it will mark the parameter receipt using `cede`:

```toka
fn devour_data(cede list: [i32]) {
    // When this function ends, 'list' and all resources within it are destructed!
}

pub fn main() -> i32 {
    auto ^arr = new [1, 2, 3]
    devour_data(cede arr) // Explicitly telling the compiler: I am severing my ownership.
    return 0
}
```

## The `drop` Method & Custom Destructor Boundaries (`@encap`)

For standard structs, the Toka compiler automatically generates a deep, cascading recursive destructor setup (for instance, all pointers contained within a Struct drop recursively without you lifting a finger!).

However, for externally managed resources (like an OS File Handle or a Network Socket), you can intercept and write your own closure logic using the proprietary `@encap` Trait:

```toka
shape File (
    handle: i32
)

// Implementing an exclusive codex for File using the @encap trait
impl File@encap {
    // Mandatory signature: must be the stealthy fn drop(self#) 
    fn drop(self#) {
        unsafe {
            libc_close(self.handle)
        }
    }
}
```
When `drop` is defined, it triggers unfailingly the moment the object's countdown hits zero.
