As your Toka applications and libraries grow, maintaining a clean and intuitive directory structure becomes essential. Toka does not strictly enforce folder layouts, but following the community-driven conventions ensures that your code remains scalable, readable, and easy for other developers to integrate.

## 1. Executable Applications (Binaries)

If you are building an application that will be compiled into an executable, the entry point should always be `src/main.tk`.

### Standard Layout

```text
my-app/
├── Project.tk        # Build system configuration
├── src/
│   ├── main.tk       # The main entry point containing pub fn main() -> i32
│   ├── utils.tk      # Application-specific logic
│   └── models/
│       ├── user.tk
│       └── session.tk
└── tests/            # Integration and unit tests
    └── pass/
```

### Best Practices for Executables
- **Entry Point**: Always place your `pub fn main()` inside `src/main.tk`.
- **Modularity**: Break down large features into subdirectories inside `src/` (e.g., `src/models/`, `src/services/`).

## 2. Libraries (Packages)

When building a library for other Toka developers to use, the organization must prioritize a **clean import experience**.

> [!WARNING]
> **Anti-Pattern: The Microservice Layout**
> Avoid creating a deeply nested directory for every single component and naming the file `main.tk`.
> 
> **Bad Example:**
> ```text
> lib/
> ├── mylib_button/
> │   └── main.tk    ❌ Do not use main.tk for library components!
> ├── mylib_chart/
> │   └── main.tk    ❌ Requires: import lib/mylib_chart/main
> ```
> This structure is overly fragmented and creates a miserable import experience for users.

### Standard Layout

Instead, flatten your library components and group them under a unified namespace directory. Name the files exactly what they represent.

```text
my-lib/
├── Project.tk
├── lib/
│   └── mylib/           # The unified namespace directory
│       ├── button.tk    # Clean, intuitive filenames
│       ├── chart.tk
│       └── colors.tk
└── tests/
```

### Best Practices for Libraries
- **No `main.tk`**: Never use `main.tk` inside library directories. `main.tk` is strictly reserved for executable entry points.
- **Unified Namespace**: Always place your library files inside a subdirectory under `lib/` (e.g., `lib/mylib/`). Do not pollute the root `lib/` directory with loose files.
- **Clean Imports**: With the standard layout, your users will enjoy clean, idiomatic imports:
  ```toka
  import lib/mylib/button::{render_button}
  import lib/mylib/chart::{draw_chart}
  ```

## 3. General Conventions

- **File Extensions**: All Toka source files must use the `.tk` extension.
- **Snake Case**: Directory and file names should use `snake_case` (e.g., `file_reader.tk`, `network_client/`).
- **Constant Extraction**: If your library defines many constants (like ANSI color codes or mathematical constants), extract them into a dedicated file (e.g., `colors.tk` or `constants.tk`) rather than intermingling them with logic.
