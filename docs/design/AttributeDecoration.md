# 📜 Toka 1.3 指针属性流转与赋值兼容性宪章 (修正终稿)

本宪章规定了在 **Copy 语义** 逻辑下，Toka 1.3 如何通过物理位置（前缀与后缀）对赋值操作（`LHS = RHS`）进行安全审计。LHS 与 RHS 在物理上相互独立，赋值过程实质上是**“值状态”对“新容器契约”的合规性检测**。

## 第一章：核心定义与分层原则

### 1.1 物理区隔

权限通过符号在声明中的位置实现物理断路：

* **前缀区 (Identity/Address Layer)**：紧跟形态符（`*`, `^`, `~`, `&`），管控地址重绑定（Reseat）与 `nullptr`。
* **后缀区 (Soul/Object Layer)**：紧跟标识符名，管控内容修改（Mutation）与 `none`。

### 1.2 属性语义映射表

| 符号 | 前缀：身份层 (Address) | 后缀：灵魂层 (Soul) |
| --- | --- | --- |
| **`$`** | **Fixed**: 必有且地址锁定。 | **Immutable**: 必存且内容只读。 |
| **`#`** | **Reseatable**: 必有但地址可变。 | **Writable**: 必存但内容可改。 |
| **`?`** | **Nullable**: 可空但地址锁定。 | **Nullable**: 可为 `none` 但内容只读。 |
| **`!`** | **Full Access**: 可空且地址可变。 | **Full Access**: 可为 `none` 且内容可改。 |

## 第二章：身份层（前缀区）兼容性规则

身份层审计决定了 RHS 的“存在性”是否满足 LHS 的“容器契约”。

### 2.1 重绑定审计 (Reseat Audit)

* **豁免状态**：处于 `unset` 的 LHS 享有初生豁免权，接受任何值的写入。
* **锁定状态**：若 LHS 已定义且前缀为 `$` 或 `?`，任何赋值操作均被编译器拦截。
* **开放状态**：仅当前缀为 `#` 或 `!` 时，允许执行重新赋值。

### 2.2 空安全准入 (Nullability Audit)

* **安全提升 (Safe Promotion)**：非空值（RHS 为 `$` 或 `#`）可存入任何前缀的 LHS。
* *逻辑*：LHS 容器容量更大，兼容非空值。


* **降级拦截 (Demotion Guard)**：可能为空的值（RHS 为 `?` 或 `!`）严禁存入要求非空的 LHS（`$` 或 `#`）。
* **修复路径**：必须通过 unwrap()` 或if-else/match 等控制流分支将 RHS 提升为非空状态的一个 RHS。如：
```toka
if cursor.~?next is cursor.~next //运行时判断,编译器不操作 RC，性能最高，其余几个从语法上需要 RC 但编译器也可以智能优化
{
	~#cursor = cursor.~next //此时直接使用 cursor.~next 是合法的语句
} 
if cursor.~?next is auto ~next //运行时判断并赋予一个新的 ~next
{
	~#cursor = ~next //此时直接使用 ~next 是合法的语句
} 
if auto ~next = cursor.~?next { ~#cursor = ~next }
match cursor.~?next {
	auto ~next => { ~#cursor = ~next }
	_ => {...}
}
```

## 第三章：灵魂层（后缀区）兼容性规则

灵魂层审计确保了“访问代理”的合法性，防止权限窃取。

### 3.1 修改权限审计 (Mutation Audit)

* **权限收敛（允许）**：可写灵魂（RHS 为 `#` 或 `!`）可存入只读 LHS（后缀为 `$` 或 `?`）。
* **权限窃取（禁止）**：只读灵魂（RHS 为 `$` 或 `?`）**严禁**存入可写 LHS（后缀为 `#` 或 `!`）。
* *逻辑*：代理人（LHS）的权利不能超过委托人（RHS）的授权。



### 3.2 存在性审计 (None-safety Audit)

* **存在性提升（允许）**：必存灵魂（RHS 为 `$` 或 `#`）可存入允许 `none` 的 LHS（后缀为 `?` 或 `!`）。
* **虚无渗透（禁止）**：可能为 `none` 的灵魂（RHS 为 `?` 或 `!`）严禁存入要求必存的 LHS（后缀为 `$` 或 `#`）。

### 3.3 灵魂层属性兼容性矩阵 (LHS = RHS)

此矩阵仅针对**后缀区**的流转。

| RHS \ LHS | **`$`** (必存/只读) | **`#`** (必存/可写) | **`?`** (可选/只读) | **`!`** (可选/可写) |
| --- | --- | --- | --- | --- |
| **`$`** (必存/只读) | ✅ 对等 | ❌ **越权** (试图修改只读) | ✅ 提升 | ❌ **越权** (试图修改只读) |
| **`#`** (必存/可写) | ✅ 收敛 | ✅ 对等 | ✅ 提升 | ✅ 提升 |
| **`?`** (可选/只读) | ❌ 降级 | ❌ 降级 | ✅ 对等 | ❌ **越权** (试图修改只读) |
| **`!`** (可选/可写) | ❌ 降级 | ❌ 降级 | ✅ 收敛 | ✅ 对等 |

## 第四章：类型推导 (`auto`) 与流转逻辑

在使用 `auto` 声明新容器时，编译器遵循“最小特权”与“安全性透传”原则。

1. **前缀推导 (Default Lock)**：默认锁定为 `$`（Fixed）。除非显式标注 `auto #` 或 `auto !`，否则新容器不可重绑定。
2. **后缀推导 (Perfect Inheritance)**：完全继承 RHS 的后缀属性。RHS 是什么“灵魂”，LHS 便是其代理。
3. **空性透传 (Safety Propagation)**：
* 若 RHS **前缀**具有可空性（`?` 或 `!`），`auto` 推导的 LHS **前缀**必须自动包含 `?`。
* 若 RHS **后缀**具有可选性（`?` 或 `!`），`auto` 推导的 LHS **后缀**必须自动包含 `?`。

## 第五章：典型审计案例 (修正版)

### 5.1 身份提升与权限越权 (The Trap)

```toka
auto ~p$ = get_data()        // p 属性: ~$p$ (必有, 必存, 只读)

// 错误示范：
auto ~!ptr! = ~p             // ❌ 编译错误！
                             // 前缀: $ -> ! (✅ 允许：新容器更灵活)
                             // 后缀: $ -> ! (❌ 禁止：试图获取只读数据的修改权)

// 正确示范：
auto ~!ptr$ = ~p             // ✅ 允许
                             // 前缀: ! (容器可重绑定/可空)
                             // 后缀: $ (内容依然只读，契约守恒)

```

### 5.2 阻止存在性降级 (None-safety)

```toka
auto ^opt? = get_option()    // opt 后缀为 ? (灵魂可能为 none)
auto ^val$ = ^opt            // ❌ 编译错误！
                             // 灵魂层降级：不能将可能为 none 的灵魂存入必存容器 $

// 修正：
auto ^val$ = ^opt??          // ✅ 允许：显式解包，若为 none 则触发 panic

```

### 5.3 阻止权限越级 (ReadOnly -> Writable)

```toka
fn handle(node: ~Node$) {    // node 为只读灵魂
    auto ~tmp# = node        // ❌ 编译错误！
                             // 越权：无法将只读灵魂 ($) 放入可写容器 (#)
}

```

### 🏛️ 总结：流转哲学

Toka 1.3 的属性流转是**非对称**的，遵循以下铁律：

1. **对于容器 (前缀)**：**允许向宽兼容**。
* *只要你准备了更大的容器（允许空、允许换），你就能装入更确定的值。*


2. **对于内容 (后缀)**：**强制向严收敛**。
* *你绝不能通过复制一个新的句柄，来获得比原句柄更高的修改权限。*



本宪章确保了无论容器如何拷贝，数据的**不可变性 (Immutability)** 与 **存在性约束 (Existence)** 在编译期是绝对闭环的。
