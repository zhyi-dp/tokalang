

Toka 所有权宪法 (The Constitution of Ownership)
第一章：二元世界 (The Binary World)

第 1 条：栈值符号 (Stack Value Symbol)

定义：所有由 auto 或 const 引入的变量名，及其占据的栈内存空间（句柄）。

推论：const 隐含 auto 的类型推导能力，故 const X 也是栈值符号。

特性：它是所有权的法定持有者。它是编译器符号表中的主角。

范例：

x (原生变量)

^p, ~p, *p (带有帽子的指针变量本身)

&p (引用变量本身)

第 2 条：灵魂对象 (Soul Object)

定义：数据实际存在的实体。

对于原生变量 (auto x=42)：栈值符号和灵魂对象合二为一（灵肉合一）。

对于指针变量 (auto ^p=new...)：灵魂对象是 Heap 上被分配的那块内存（灵肉分离）。

脱帽 (Unhatting)：

解引用操作（如 p 相对于 ^p）是指向灵魂对象的操作。

灵魂本身不具备符号属性，它只是被栈值符号所指涉的资源。

第二章：所有权等级 (Hierarchy of Ownership)

根据栈值符号（帽子）的形态，对灵魂对象拥有不同的统治权：

栈值符号	权利等级	灵魂状态 (Soul State)	备注
const C	永久独占	Immutable (不可变)	灵魂不可被修改，符号不可被重绑定。
auto x	完全拥有	Owned (有主)	灵肉合一，生杀予夺。
auto ^p	独占代理	Owned (有主)	编译器维护“有主”状态（静态标志），运行时零开销。

第三章：移动协议 (The Move Protocol)

第 3 条：交易定义

表达式 auto lhs = move rhs 定义为一次**“完美的权利交割”**。

前提：lhs 是新生的栈值符号；rhs 必须是已存在的栈值符号，(const 变量不可 move) 。

帽子规则：对于指针情形，lhs 与 rhs 的指针形态必须兼容（通常是同类帽子，或特权升级）。

第 4 条：交易效力 (The Effect)

位拷贝 (Bitwise Copy)：rhs 栈上的位模式（指针地址或数据值）原样复制给 lhs。

符号失效 (Symbol Invalidation)：

编译器视角：rhs 符号即刻标记为 失效 (Invalid/Dead)。

代码视角：rhs 不可再被访问。

灵魂无感 (Soul Indifference)：

独占灵魂 (^)：不变更任何标志（因为主人只是换了名字，还是独占）。

共享灵魂 (~)：不触碰 RC 计数（交接班不需要通知计数器，+1 -1 = 0）。

原生灵魂：物理位置迁移，旧地址作废。

第 5 条：符号的轮回 (Reincarnation)

Shadowing：已失效的 rhs 符号名，可以通过 auto rhs = ... 再次声明，开启一段全新的生命周期（这不仅是复活，更是转世）。



第四章：复制协议 (The Copy Protocol)

第 1 条：生效时机 当表达式 auto lhs = rhs 出现，且未显式使用 move 关键字时，触发本协议。

第 2 条：共有基准动作 (The Base Action) 无论何种类型，首先执行 位模式复制 (Bitwise Copy)：将 rhs 栈值符号所占据的内存位（无论是数据还是指针地址），原样拷贝到 lhs 的栈空间。

第 3 条：附加动作 (Side Effects) 基于 rhs 的帽子形态，执行以下特定的附加逻辑：

独占指针 (^)：协议突变。

触发隐式移动。rhs 栈值符号立即失效（转入 Undefined 状态）。

注：这是唯一的“虽写做复制，实则为移动”的特例。

共享指针 (~)：权益稀释。

原子操作：指向的灵魂对象控制块 rc (引用计数) 加 1。

lhs 与 rhs 共享同一个灵魂。

原始指针 (*)：无。

仅保留基准的位拷贝，不触发生命周期变更。

引用符号 (&)：借用延伸。（本身引用是一种能力而不是真实存在的引用，比如灵魂对象 y#的&y和&y#，他们只能出现在rhs）

不可变引用 (&)：可以存在多个，灵魂对象的静态 Reader 计数 加 1。auto &a = &y；auto &b = &y；

可变引用 (&p#)：该可变引用存在`时期内`：不可存在其他任何活跃引用，且该可变引用成为当前使用灵魂对象的唯一通道。

灵魂对象 (Value)：深浅分流。

非 NeedDrop（纯数据）：无附加动作。基准位拷贝即完成了逻辑复制。

NeedDrop（含资源）：禁止隐式复制。

动作：编译器报错。

解法：用户必须显式调用 rhs.clone() 来创建新副本，或使用 move 转移所有权。



第五章：死亡协议 (The Death Protocol)

第 1 条：自然死亡 (Natural Death)

当栈值符号越过其定义的作用域（Scope）边界时，判定为自然死亡。

物理行为：系统自动回收其占据的栈内存（pop stack）。

遗嘱执行：在物理回收发生前，根据符号的帽子类型，严格执行以下协议。

第 2 条：原生灵魂的陨落 (Native Soul)

适用：原生栈变量（如 auto x = Shape(...)）。

执行顺序（洋葱模型）：

本尊清理：若实现了 drop 方法，立即调用 x.drop()。

成员清理：在本尊 drop 执行完毕后（注意：不是之前，以确保 drop 体内能安全访问成员），编译器自动按成员声明的逆序，递归调用所有成员的死亡协议。

平凡跳过：若某层级未实现 drop 且无 NeedDrop 成员，编译器跳过该步骤（零开销）。

第 3 条：独占代理的交割 (^ Unique)

适用：auto ^p。

执行逻辑：

判空：(若编译器无法静态确定非空) 检查指针是否为 null。

清理：调用指向灵魂的 drop 方法。

释放：调用 free 归还堆内存。

注：所谓的“有主状态标记”完全由编译器在静态分析阶段维护，运行时只管杀，不问标记。

第 4 条：共享代理的离场 (~ Shared)

适用：auto ~p。

执行逻辑：

减权：原子操作 rc--。

核销：检查 rc 是否归零。

若不为 0：仅离场，不做其他操作。

若为 0：触发 Final Drop（执行灵魂 drop + free 内存）。

第 5 条：原始观测者的无视 (* Raw)

适用：auto *p。

执行逻辑：什么也不做 (No-Op)。

原始指针不承担任何生命周期责任。

第 6 条：借用锁的释放 (The Release of Borrow Locks)

适用：auto &p = &origin 或 auto &#p = &origin。

核心概念：引用本身没有生命，它只是 origin 身上的一把锁。引用的死亡，意味着锁的归还。

6.1 不可变引用 (&p) —— 读锁释放 (Reader Lock Release)

生前状态：origin 身上挂着计数器 Readers += 1（静态计数）。此时 origin 不可写，但可被其他 & 借用。

死亡时刻：

编译期行为：编译器将 origin 的静态 Reader 计数减 1。

后果：若计数归零，origin 恢复完全自由身（可写、可 Move）。

运行时行为：无代码生成（Zero Cost）。

6.2 可变引用 (&#p) —— 写锁释放 (Writer Lock Release)

生前状态：origin 身上挂着独占写锁 (Exclusive Lock)。此时 origin 既不可写也不可读（被 &#p 独占）。

死亡时刻：

编译期行为：编译器移除 origin 的独占锁标记。

后果：origin 瞬间解冻，恢复可读可写状态。

运行时行为：无代码生成。
