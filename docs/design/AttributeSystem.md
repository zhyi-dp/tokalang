# 🏛️ Toka 1.3 权限位阶宪法 (最终定稿版)

本宪法规定了 Toka 语言中指针身份（Identity）与对象灵魂（Soul）的权限隔离规则、初始化豁免逻辑及成员属性继承机制。

---

## 第一章：空间分割原则 (Space Separation)

指针与对象的权限由其符号在声明中的**物理位置**严格界定。两区权限完全独立，互不渗透，无继承关系。

1. **形态前缀区 (Morphology Zone)**：紧跟在形态符（`^`, `*`, `~`, `&`）之后。
* **职能**：管控 **重绑定 (Reseat)** 权。决定指针地址（容器本身）是否可变更、是否可为 `nullptr`。
* **限制**：此 Zone 仅对四种指针形态有意义。


2. **标识符后缀区 (Identifier Zone)**：紧跟在变量名之后。
* **职能**：管控 **内容修改 (Mutation)** 权。决定对象本身或指针指向的对象（灵魂内容）是否可修改、是否可为 `none`。



> **Nullable 声明**：表示该维度可能“不存在”。`none` 与任何具体值均不相等；在指针维度，`nullptr` 为该维度的 Nullable 表现。

---

## 第二章：属性权限语义表 (Attribute Semantics)

| 符号 | 指针形态后 (Address Layer) | 变量标识符后 (Soul Layer) |
| --- | --- | --- |
| **`$`** | **Fixed**: 地址终身锁定，不可重绑定且不可为 `nullptr`。 | **Immutable**: 灵魂只读，不可修改内容且不可为 `none`。 |
| **`#`** | **Reseatable**: 地址可换，**但不可为 `nullptr**`。 | **Writable**: 内容可改，**但不可为 `none**`。 |
| **`?`** | **Nullable**: 地址锁定，**但可以为 `nullptr**`。 | **Nullable**: 内容只读，**但可以为 `none**`。 |
| **`!`** | **Full Access**: 可重绑定且可为 `nullptr`。 | **Full Access**: 可修改内容且可为 `none`。 |

注意：不能用#?或者?#的双符号来代替!符号，我们只有一个符号位置来承载他们
---

## 第三章：初生豁免权 (Initialization Exemption)

1. **定义**：**初始化 (Initialization)** 不属于重绑定或修改内容。无论是出生时的原地初始化，还是从 `unset`（物理未定义）状态填入第一个有效值（含 `nullptr/none`）的过程，均享有豁免权。
2. **规则**：
* 若变量处于 `unset` 状态，首次赋值无须其属性具备 `#` 或 `!` 权限。
* 一旦首次赋值完成，状态坍缩为 `defined`。后续任何变更必须严格匹配声明时的 `#` 或 `!` 权限。
* **前置约束**：若预期后续需要修改，则定义时必须具备相应权限，豁免仅对“第一次写入”有效。



---

## 第四章：断路原则 (The Disconnection Law)

权限在不同平面（指针  灵魂）之间**严禁自动渗透**：

1. **重绑定断路**：拥有 `^#p`（地址可变）不代表拥有内容修改权。
2. **内容修改断路**：拥有 `^p#`（内容可写）不代表拥有重绑定权。
3. **独立校验**：改地址查前缀，改内容查后缀。

---

## 第五章：成员访问与权限继承 (Member Access)

针对 Shape 成员，权限继承仅针对 **可写性 (#/!)**，不涉及可空性。

1. **直接成员 (Direct Member)**：Shape 字段本身（如 `val: i32` 或 `^ptr: Node`）。
* **继承规则 1 (默认继承)**：若无任何属性标记，可写性随父容器实例的可变性动态变化。
* **继承规则 2 (显式覆盖)**：若标注了属性，则无视父容器权限。用于实现**内部可变性**（即使父容器只读，成员也可显式设为可写）。
* **继承规则 3(嵌套继承)**: 对于直接成员，继承是递归嵌套继承的。指针指向的对象不属于直接对象，遵守断路原则，继承终止。


2. **间接灵魂 (Indirect Soul)**：通过指针成员解引出的对象。
* **继承规则 4**：不参与继承逻辑。不标注时默认等价于 `$`（只读）。


3. **作用域父容器假说**：
* 独立变量的父容器被视为“作用域”。作用域的权限始终为**只读**。
* 推论：不标注属性的独立变量默认继承作用域的只读属性，实现了“变量默认不可变”的语言特性。

---

## 第六章：指针重绑定后的“原物处理”准则

当发生 Reseat（重绑定）操作导致指针指向新地址（或 `nullptr`）时，原地址对应的资源处理如下：

* **原始指针 (`*`)**：**不处理**。编译器仅更新地址，若涉及堆内存释放，需由开发者手动完成。
* **独占指针 (`^`)**：**自动清理**。重绑定后，原灵魂对象立即处于无主状态，由编译器插入析构与释放逻辑。
* **共享指针 (`~`)**：**引用计数递减**。原灵魂对象的 RC（Reference Count）减 1。
* **引用 (`&`)**：**不处理**。借用者不持有所有权，不影响原物的生命周期。

---

## 第七章：Sema 强制审计逻辑

1. **Reseat 审计**：执行 `LHS = RHS` 时，校验 LHS **形态前缀** 是否包含 `#` 或 `!`。
2. **Mutation 审计**：执行 `LHS.field = val` 或 `LHS# = val` 时，校验 LHS **变量名后缀** 是否包含 `#` 或 `!`。
3. **空值审计**：
* 写入 `nullptr`  校验前缀是否包含 `?` 或 `!`。
* 写入 `none`  校验后缀是否包含 `?` 或 `!`。



---

## 第八章：指针判空与逻辑类型提升 (The Six Paths)

当对 `^?` 或 `^!` 等具有可空属性的指针进行操作时，必须通过以下方式之一提升其位阶至 `^`（非空状态）：

1. **DFA 智能探测**：`if (^?p != nullptr) { ... }` —— 块内 `p` 自动提升为非空。
2. **穷举匹配**：
```toka
match ^?p {
    ^q => { ... } // 自动解包绑定为非空标识符 q
    _  => { ... } // 处理 nullptr
}

```


3. **影子绑定**：`if auto ^q = ^!p { ... }` —— 产生临时的非空替身 `q`。
4. **身份谓词**：`if ^?p is ^p { ... }` —— 显式测试身份层是否具备实体。
5. **显式断言**：`auto &r = &(p.unwrap())` —— 程序员显式声明对风险的承担。运行时判断，如果为空则 panic
6. **后缀解包算子**：`auto &r = &(p??)` —— 快捷解包算子。
* *注：为避免 `???` 冲突，当表达式结尾本身带有 `?` 属性符时，编译器解析器强制要求使用空格或括号分隔。*



---

### 🛡️ 最终安全性评估

1. **符号唯一性**：通过第二章的强约束，保证了编译器在解析每一个 Token 时，对权限的判定是常数时间  的。
2. **解包与提升**：第八章规定了“六脉神剑”，确保了在 Toka 1.3 中，没有任何一个空指针能够逃过语义审计。
3. **数据流闭环**：结合 `.unset()` 的 DFA 分析与 `.unwrap()` / `??` 的显式解包，内存的“生死状态”在编译期达到了 100% 可视。

---

贤弟，这版宪法已经把 **“符号位阶的极简性”** 和 **“逻辑提升的严谨性”** 统一起来了。尤其是限制了双符号的使用，这不仅让语法更清爽，更让编译器后端的类型推导（Type Inference）变得极其稳固。

额外记录：
关于 unset（延迟初始化），为了减轻编译器的负担，我们在 unset 状态的变量真实初始化时使用.unset()标注
```toka
auto ^head = new Node(val=101, ^?next=unset) 
head.^next.unset() = new Node(val=101, ^?next=unset) //运行时判断 如果他不是 unset 的，则 panic
```
