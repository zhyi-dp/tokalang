# Toka 泛型系统最终宪法 (Final Specification v0.6.0)

## 1. 核心哲学：虚实二分与作用域隔离

Toka 泛型系统建立在**定义期（Definition）**与**实例化期（Instantiation）**的绝对物理隔离之上，以彻底消除系统级编程中常见的抽象混淆。

* **泛型形参 (Generic Formal Parameters / Sentries)**：
* **定义**：带有阶序哨兵（Rank Sentry）符号的抽象占位符。
* **作用域**：**仅限**存在于 `shape`、`trait` 或 `fn` 的 `<...>` 声明列表及其随后的定义体内。
* **职责**：通过物理形状宣告该坑位预期的维度规格。


* **泛型实参 (Generic Actual Arguments / Concrete IDs)**：
* **定义**：已定义的具体类型标识符（如 `i32`, `Vec`, `OptionTrans`）。
* **要求**：在实例化点（Usage Site）使用，**严禁**带有任何前缀哨兵符号。其阶序（Rank）在定义时已作为元数据固定。



---

## 2. 递归阶序定律 (The +1 Law of Rank)

具体类型的阶序是由其定义的**泛型形参**中阶序最高者递归决定的。它是衡量类型抽象深度的唯一量纲。

* **0 阶基础 (Base Case)**：不带泛型参数的类型（如 `i32`, `f64`）或 0 阶形参本身即为 **0 阶**。
* **1 阶构造子**：形参最高为 0 阶（如 `shape Vec<T>`），则 。
* **高阶构造子**：形参最高为  阶（如 `shape MonadTrans<2_M>`），则 。

---

## 3. 阶序哨兵法则 (The Rank Sentry Rules)

在 `<...>` 语境中，形参的物理构造决定其阶序。**标识符主体部分必须以大写字母 [A-Z] 开头（PascalCase）。**

| 形参阶序 | 维度本质 | 哨兵模式 (Pattern) | 示例 | 预期实参阶序 |
| --- | --- | --- | --- | --- |
| **-1 阶** | 常量 (Value) | `ID + _` (后缀) | `N_`, `Size_` | 具体数值 (Value) |
| **0 阶** | 类型 (Type) | `ID` 或 `0_ID` | `T`, `Item`, `0_T` | **0 阶** 具体类型 |
| **1 阶** | 1 层构造子 | `_ + ID` 或 `1_ + ID` | `_V`, `_List`, `1_V` | **1 阶** 类型构造子 |
| **2~9 阶** | N 层构造子 | `Digit + _ + ID` | `2_M`, `3_Monad` | **N 阶** 类型构造子 |

---

## 4. 阶序三铁律 (The Ironclad Rules)

1. **物理隔离律**：哨兵位前缀（`_` 或 `Digit_`）后必须紧跟**大写字母**。这确保了它与数字分隔符（`1_000`）、带单位字面量（`10_ms`）或局部变量（`_val`）在词法上绝对隔离。
2. **前缀规范律**：`Digit` 严格限制为单字符 `0-9`（即最高支持 9 阶）。`_T` 与 `1_T` 等价；`T` 与 `0_T` 等价。系统推荐简洁风格，但支持显式对称写法以满足严谨化需求。
3. **哨兵独占律**：数字开头加下划线的模式（如 `2_M`）被泛型声明语境独占，严禁作为普通变量名或具体类型名使用。

---

## 5. 维度传递与填坑准则 (Dimension Conservation)

实例化是使用**实参**填充**形参**的过程，遵循“阶序强匹配”与“维度减法”。

### 5.1 强匹配准则 (Rank Alignment)

在实例化点，传入实参的具体  必须严格等于该位置形参定义的哨兵 。**禁止跨阶填充**（例如：2 阶槽位不能接受 0 阶实参）。

### 5.2 填坑降维 (Dimension Subtraction)

填入一个参数意味着消除一层抽象。公式如下：


* **2 阶路径**：形参 `2_M` (Rank 2) 被 2 阶实参填充后，结果在该作用域降级为 **1 阶** 语义。
* **1 阶路径**：形参 `_V` (Rank 1) 被 1 阶实参填充后，结果在该作用域降级为 **0 阶** 语义（即具体类型）。

## 6. 编译器实现指南 (Compiler Implementation)

### 6.1 Lexer 确定性扫描 (Lookahead 2)

在 `<...>` 泛型环境内，通过预读前两个字符实现  无回溯判定：

1. **判定 2~9 阶**：`peek(0)` 为 `[2-9]` 且 `peek(1) == '_'` 且 `peek(2)` 为 `[A-Z]`。标记为 `TOKEN_GENERIC_FORMAL_N`。
2. **判定 1 阶**：`peek(0) == '_'` 且 `peek(1)` 为 `[A-Z]`。标记为 `TOKEN_GENERIC_FORMAL_1`。
3. **判定 -1 阶**：标识符以 `_` 结尾。标记为 `TOKEN_GENERIC_FORMAL_NEG1`。
4. **判定 0 阶**：标识符以 `[A-Z]` 开头，且不满足上述条件；或显式以 `0_` 开头。标记为 `TOKEN_GENERIC_FORMAL_0`。

### 6.2 Sema 维度核查

* **元数据绑定**：每个类型在解析定义后，其  写入符号表。
* **匹配逻辑**：在处理实例化点（Call Site）时，Sema 执行 `Actual.Rank == Formal.Rank` 的整数比对。若失败，立即抛出 `RankMismatchError`。
* **作用域清理**：哨兵标识符在离开其定义的声明体作用域后立即销毁。
